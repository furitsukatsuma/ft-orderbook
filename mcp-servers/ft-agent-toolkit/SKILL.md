# ft-agent-toolkit

ft-automation の AI コンビニ向け**現場代理人スキル**です。お客 AI が購入・交渉・検品を行うための**3 層ネゴシエーター構造**（意図確認 → 範囲交渉 → 実行契約）を支援します。MCP 実装は `src/index.ts` を参照してください。

## セキュリティ対策

交渉フローや顧客入力を扱うスキル／ツールでは、次を**必須**とします。

### SQL インジェクション

- **文字列は「プレースホルダ + バインド」**で渡す。ユーザー入力をクエリに直挿入しない。例: `better-sqlite3` の `prepare` と `?`（または名前付きプレースホルダ）。
- **手書きの文字列エスケープだけに頼らない**。全角文字・Unicode・コメント混入などで抜け道が出やすく、主対策にはなりません。
- **ユーザー由来の文字列を SQL リテラルへ連結しない**。テーブル名・列名・`ORDER BY` など動的識別子は、可能なら**ホワイトリスト**で固定化する。

次のような単純エスケープは**参考程度**に留め、**SQL 防御の中心にしない**こと。

```javascript
function escapeSQL(input) {
  return input.replace(/'/g, "''");
}
```

### プロンプトインジェクションについて

`escapePrompt` のような単純な置換だけでは、**指示の上書き**や**悪用**を防ぎ切れません。**多層防御**で設計します。

- **ユーザー原文をシステム指示に直埋めしない**（必要なら要約・構造化フィールドのみ）。
- **区切りと役割の分離**（ユーザーメッセージとシステム方針を明確に分ける）。
- **許可リスト**でツール・操作・出力形式・パスを制限し、逸脱を拒否する。
- **ツール実行前の検証**（危険なコマンド・パス・外部 URL は実行前に拒否）。
- **リスクの高い操作**は実行前確認・ログ・人間レビューを組み合わせる。

### 補足

- `escapeSQL` / `escapePrompt` を使う場合は、**主対策か補助か**と適用範囲を文章で明記する。主対策はバインドと設計分離。
- **入力検証**と、危険操作に対する**デナーリスト／ポリシー**（シェル直実行・任意 SQL 実行・ファイル削除系などを拒否）を併用する。

### レビューとテストの必要性

- **コード変更はレビュー**（セキュリティ観点を含む）を通す。
- **自動テスト**に加え、必要に応じて**手動のセキュリティ確認**（高リスクフロー、ツール境界）を行う。
